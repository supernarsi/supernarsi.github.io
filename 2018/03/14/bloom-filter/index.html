基于 Bloom Filter 的新闻去重

发表于 2018-03-14 分类于 技术 阅读次数： Disqus：
这段时间在看一些算法相关的知识，看到 Bloom Filter 时，想起以前做的新闻项目，为解决已读新闻的去重问题，曾用到了该算法，但有些细节比较模糊了，于是回顾了一下当时的解决思路，顺便总结整理了下当初考虑不足的地方，在此做个简单的笔记。

这段时间在看一些算法相关的知识，看到 Bloom Filter 时，想起以前做的新闻项目，为解决已读新闻的去重问题，曾用到了该算法，但有些细节比较模糊了，于是回顾了一下当时的解决思路，顺便总结整理了下当初考虑不足的地方，在此做个简单的笔记。

关于 Bloom Filter 的介绍，网上很多资料，就不再赘述了。要点就是结合哈希算法和 bitmap 的数据结构，在可容忍一定错误率的前提下，实现比较高效的数据判重，同时又尽可能的节省内存空间占用。

我们之前项目遇到的情况是，爬虫每天会采集到大量的新闻存入库，如果要上推荐系统，用户就可能在不同的栏目（推荐分类和其他细分分类）下，看到重复的新闻，于是需要一个新闻去重的模块，在产出新闻队列前，先将用户看过的新闻过滤掉。

如果单纯的新闻去重，即使千万量级直接存在数据库或缓存里查问题也不大，但如果要做的是针对每个用户做已读的去重，那这个已读集合的数量就变成了新闻数乘以用户数，量级将大得惊人。加之由于是 API，对响应时间也有一定要求，因此去重逻辑不能太耗时。于是我们考虑基于 Bloom Filter 来实现这个需求：将用户 id + 新闻 id 做为输入字符串进行去重判断。

但是，如果直接简单的套用 Bloom Filter 做判重，也还是存在很大问题。做个简单的算术：假设有 10w 活跃用户（其实当时我们只有 1w 左右的 DAU），可读新闻有 10w，要判断某个用户是否读过某篇新闻，需要判重的集合达到 100,000 x 100,000 = 10,000,000,000 的级别，如果允许的错误率为 5%（即每 100 篇新闻，可能至多有 5 篇没有读过的新闻被判断为已读），通过 Bloom Filter 的公式可以根据需要元素 n 和容错率 p 推算出所需 bit 位的大小 m（详细推导可参考该篇 文章）：

$$m = -\frac{(n * ln(p))}{(ln(2))^2} = 95850583773bit ≈ 11.16G$$

需要 11G 的内存几乎是不可接受的，于是我们回过头来重新分析业务需求。首先新闻是有时效性的，无论是在推荐中还是细分类别中的新闻，较新的新闻权重都会更高，因此用户看到几天前新闻的概率其实是比较低的，其次，每个用户看新闻的量也是可评估的，正常情况下用户不可能无限次的获取新闻。

结合以上两点，我们重新评估了数据量：假设有 10w 活跃用户，每个用户每天最多看 1000 篇新闻，那么其实每天需要判重的数据集就减少到了 100,000 * 1000 = 100,000,000，使用 Bloom Filter 处理的话需要的内存空间也变成了 623522422bit ≈ 74M。

但不可避免地，还是会有用户可能看到昨天甚至前几天的新闻，特别是对于推荐这个分类来说，时效性相对较弱，近两天的新闻还是可能会被推荐。那如何解决今天之外新闻的去重？每天一个 bitmap 不就好了。每篇新闻有自己的产生时间，以这个时间的对应日期，来做 bitmap 的拆分，最多保留七天的 bitmap（一周以前的新闻几乎已经没有意义了），占用内存也只有大约 520M 而已，完全在可接受范围。

当时我们的存储直接使用了 Redis 的 bitmap 数据结构，值得一提的是 Redis 的单个 bitmap 至多支持 2^32bit，好在我们拆分到每天做去重后，每个 bitmap 只用分配几十 M，完全够用了。

回顾完之前的做法后，想到了一些存在问题：

  1. 之前使用的 MurmurHash 函数是使用网上已实现的版本，但只支持 32 位的 hash，如果我们需要分配的 bit 位大于了 2^32，那么将有一部分空间无法被哈希到，间接会导致错误率提高。当然此时 Redis 的存储也会有问题。
  2. 我们的实现是基于 10w 用户，每位用户至多 1000 篇新闻这个数据假设的，但如果有用户阅读超过单天 1000 篇甚至更多新闻，其实就占用其他用户的可读空间。总体来看的话，只要单日去重的数据集超过了 1 亿，那么判重错误率将显著上升（因为 bit 位会逐渐被占满）。
  3. 类似第 2 点，由于我们没有做 bitmap 的写入限制，一旦用户激增或者有人恶意刷新闻阅读，bitmap 一定会达到每一位都置为 1 的情形，那时判重错误率将达到 100%。
